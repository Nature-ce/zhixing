# Glossary

## 任务（Task）

用户输入的顶层目标，例如"完成读书笔记"、"筹备婚礼"。任务是整个工作流的起点，由 AI 拆解为若干子项目。

### 任务字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `标题` | text | 是 | 任务名称 |
| `描述` | text | 否 | 补充说明 |
| `状态` | enum | — | `进行中` / `已完成` / `已放弃` |
| `目标完成日期` | date | 否 | 计划什么时候做完，为日程排期提供边界约束 |
| `创建时间` | timestamp | — | 自动记录 |
| `完成时间` | timestamp | — | 状态变为已完成/已放弃时自动记录 |
| `回顾` | text | — | 完成后的纯文本总结，与任务 1:1 |

### 计划层与排期层的一致性（Cascade 规则）

子项目（计划层）发生改动时，日程（排期层）按以下规则自动联动，保持两层一致：

| 改动场景 | 日程层联动行为 |
|----------|----------------|
| 编辑已排期子项目的标题或预估时长 | 日程格子里的标题/时长 **自动跟着变**（同一实体，引用关系） |
| 删除已排期的子项目 | 对应日程格子 **自动清空** |
| 任务被标记为"已放弃" | 该任务下 **所有子项目自动变为已放弃**，且对应日程格子 **全部清空** |
| 把子项目拖放到已被占用的时间格子（**同一任务内**） | 系统 **拒绝放置**（提示冲突），不允许同一任务内日程重叠 |
| 把子项目拖放到已被占用的时间格子（**跨任务**） | **允许重叠**，日程视图同时显示来自不同任务的子项目，用任务色彩区分来源 |

### 逾期处理

子项目已排期的时间已过时，**状态保持"已排期"，不引入新状态**。视觉上做"过期"弱化显示（灰色淡化 + 显示"已过期"小标签）。操作上用户仍可手动完成、放弃或改期。

### 回顾可编辑性

回顾**可反复编辑**，无锁定机制。用户过几天仍可回来补充。

### 搜索

MVP 提供**全局搜索**：用户可按关键词搜索任务和子项目。随任务量增长，这是高频刚需。

### 数据导出

MVP 提供**数据导出**：由于是纯本地单机（无云端备份），导出是用户最重要的数据安全感来源。导出格式待定（JSON 或 Markdown）。

### 任务信息修改

任务创建后，**标题和描述都可以反复编辑**，无锁定。创建后调整任务信息是常见操作。

### 子项目跨天

子项目**允许跨天**：一个子项目可以占据连续多天的时间段（如周日晚 8-10 点 + 周一晚 8-10 点）。日程视图上同一子项目在多天时段中呈现为关联的一组分块。

### 回顾 tab 结构

- **列表层**：按任务归类，每个任务只显示标题，其下展示该任务已完成/已放弃的子项目条目。用户一眼看到"哪些任务完成了、每个任务下有哪些子项目"
- **任务详情层**：点击某个任务进入该任务的完整回顾页，页面分上下两块：
  - **上方**：进度图——展示整个任务的全貌，包含每个子项目的名称、完成日期，以及整体进度
  - **下方**：总结内容——该任务的回顾纯文本

### 日程提醒

MVP **不做推送提醒**。子项目排期后，日程仅作为"计划看板"，用户主动打开 app 查看今天做什么。不引入本地通知权限、后台调度、逾期提示等功能。后续基于用户反馈决定是否在 V2 加入。

### 数据持久化

MVP **纯本地单机**：所有业务数据（任务/子项目/日程/回顾）存在 Android 手机上，使用 Room 数据库。无账户、无登录、无同步。换手机/重装 app 数据丢失。后续按需可升级为"本地为主 + 可选云备份"。

### 空状态

- **第一次打开 app（无任何数据）**：一个干净的欢迎页，中间一句欢迎/价值主张文案（如"把大任务拆成小项目，一步步完成"），下方一个醒目的"创建第一个任务"按钮。不做多页 onboarding 滑动页
- **任务列表为空**：插画/图标 + 一句引导文案 + 创建按钮
- **日视图当天无排期**：空白时间轴 + 一句轻引导文案，不破坏日视图结构

### 设计风格

MVP 遵循**清新简约**原则，可执行约束如下：

- **色彩**：大面积白色/浅灰底色 + 一种低饱和主色（如薄荷绿、雾蓝）作为强调；不引入多彩色或强对比色
- **排版与留白**：大留白、大字号、卡片大量留空；信息密度让位于呼吸感
- **动效**：克制——只用淡入淡出、微小位移等轻反馈；不用弹性动画或复杂转场

这些约束作为 UI 设计与组件选择的边界条件，任何新增视觉元素都需先过这一关。

### App 导航结构

MVP 采用底部 **tab bar 一级导航**，三个 tab：

- **任务**：任务列表主页，创建和管理任务
- **日程**：默认打开日视图，可切换到周视图；含 backlog 面板
- **回顾**：回看所有已完成/已放弃任务的历史总结

**启动落地页**：app 启动后默认进入「日程」tab 的日视图，用户第一眼就看到"今天做什么"。

### 任务列表

- **卡片内容**：极简——标题 + 状态标签 + 进度数字（如"3/8"）
- **排序**：按创建时间倒序
- **已完成/已放弃任务**：折叠到列表底部一个"已完成"分区

任务详情内部按"子项目列表 → 日程排期 → 回顾（完成后解锁）"组织，从任务列表点进任务详情。

### 任务状态流转

- 创建后默认 `进行中`
- `进行中` ──(所有子项目进入终态)──► `已完成`（自动触发，并解锁回顾）
- `进行中` ──(用户手动放弃)──► `已放弃`
- `已完成` / `已放弃` 是终态

### 任务删除

MVP 支持**彻底删除任务**，级联删除其下所有子项目及对应日程占用。执行前必须经过确认弹窗。"已放弃"（保留痕迹）和"彻底删除"（不留痕迹）两种心理需求都给予满足。

## 子项目（Subproject）

AI 对任务拆解后产生的计划实体，是本应用**拆解的最小单元**。子项目拥有独立的生命周期状态（见下方"子项目状态"）。子项目与日程格子是**两层分离**的关系——子项目是"做什么"，日程是"什么时候做"。子项目可以暂时不排期（停留在 backlog），也可以改期、跨天。

### 子项目字段

MVP 阶段子项目仅包含以下字段：
- `标题`（text）—— 做什么
- `状态`（enum）—— 见"子项目状态"
- `预估时长`（duration）—— 这件事大概做多久，影响日程安排

早期不引入优先级、依赖关系、标签/分类。优先级和依赖关系靠日程的先后顺序隐式表达。

## 子项目状态

子项目在生命周期中经历四个状态：

| 状态 | 含义 | 进入条件 |
|------|------|----------|
| **backlog** | 已拆解但尚未放进日程 | AI 拆解后默认进入 |
| **已排期** | 放进日程格子、完成前 | 用户手动放置到时间轴 |
| **已完成** | 做完了 | 用户手动标记（可从 backlog 直接跳，也可从已排期进入） |
| **已放弃** | 主动放弃，不做 | 用户手动标记 |

### 流转规则

- backlog ──(放置到日程)──► 已排期
- backlog ──(标记完成)──► 已完成（允许跳过排期直接完成）
- backlog ──(放弃)──► 已放弃
- 已排期 ──(标记完成)──► 已完成
- 已排期 ──(放弃)──► 已放弃
- 已完成 / 已放弃 是终态，但 MVP 允许**误操作撤销**（已完成可改回已排期）。目的是避免误标完成时被迫"删掉重建"

### 任务完成条件

一个任务进入"已完成"，当且仅当它的**所有子项目都进入终态**（已完成 或 已放弃）。只要还有子项目停留在 backlog 或 已排期，任务就仍为"进行中"。任务完成后解锁"回顾"。

### AI 拆解行为

AI 拆解是**一次性生成**动作：用户输入任务 → loading → 返回完整子项目列表。生成后用户可在列表里手动增删改（标题、预估时长），但不再追问 AI 做二次拆解。

### 拆解调用链路

app 不直连大模型，而是请求自建后端服务，由后端持有 API Key 并转发给大模型。这是架构决策，理由见 `docs/adr/0001-ai-via-backend-proxy.md`。

### 模型选择

MVP 后端对接 **DeepSeek** API（性价比高、中文任务拆解质量好、有稳定 OpenAI 兼容接口）。后续可按需切换或增补其他提供商。

## 日程表（Schedule）

对子项目的**排期层**，把子项目放进具体的日期/时间段。日程表是子项目在时间轴上的投影，不改变子项目本身的状态。

### 排期交互

MVP 采用**手动放置**作为主体交互：用户在日程视图（时间轴）上手动把子项目拖进某个时间段，或点击空格子指派一个 backlog 里的子项目。用户完全控制"什么时候做什么"。子项目可以留在 backlog 中暂不排期。

### 日程视图形态

- **默认日视图为主体**：用户打开日程首先看到当天的时间轴，符合"今天做什么"的每日节奏
- **提供视图切换按钮**，可切到周视图查看本周全貌
- 时间格子最小粒度 MVP 阶段推荐 **30 分钟**
- 未排期 backlog 作为日程页内常驻面板，支持在同一个画面完成"看日程 → 找 backlog 里的事 → 拖进去"的闭环

## 回顾（Review）

任务完成后的复盘动作。

### 内容结构

MVP 采用**纯文本单体总结**。任务完成后解锁"写回顾"入口，用户自由撰写一段总结体会（学习心得、项目复盘等），无强制模板。

### 触发时机

严格在任务状态变为"已完成"之后才解锁。任务进行中的过程里不可写回顾。

### 存储关系

回顾挂在**任务**上（而非子项目上）。每个任务最多一篇回顾。保存后可在任务详情里随时回看。
